## Ⅰ. 사업 개요

### 1. 사업 목적

본 사업은 사내 개발자들의 자기주도 학습 문화를 정착시키고, 학습 과정을 가시화하여 동기를 부여하기 위한 **학습 트래커 + 미니 블로그 플랫폼**("DevLog")을 구축하는 것을 목적으로 한다. AI 기반 스펙 주도 개발(Spec-Driven Development) 방법론을 적용하여, 본 명세서만으로 처음부터 구현 가능한 수준의 결과물을 산출한다.

### 2. 사업 개요

| 항목 | 내용 |
|------|------|
| 사업명 | DevLog — 개발자 학습 트래커 및 미니 블로그 플랫폼 구축 |
| 추진 방식 | AI 스펙 주도 개발 (spec-kit, Claude Code 등 활용) |
| 대상 사용자 | 사내 개발자 (기술 이해도 높은 동료 그룹) |
| 운영 환경 | 사내망 (외부 인터넷 의존성 최소) |
| 사용 언어 | 한국어 단일 언어 |
| 핵심 가치 | 학습 기록의 시각적 누적 / 가벼운 블로그 / 사내 데모 완결성 |
| 배포 형태 | Next.js Standalone 빌드 + PM2 프로세스 관리 |

### 3. 추진 배경

- 사내 개발자의 학습 활동을 가시화할 수 있는 도구의 부재
- GitHub Contribution Graph와 같이 누적이 보이는 시각적 동기부여 위젯에 대한 수요
- 동료 간 학습 콘텐츠 공유를 위한 경량 블로그 채널 필요
- 사내망 환경에서 외부 의존성 없이 완결되는 데모 시스템 요구
- AI 기반 개발 방법론(Spec-Driven Development)의 사내 실증 사례 확보 필요

### 4. 사업 내용

- 글 작성·조회·검색·필터링·정렬 기능을 갖춘 미니 블로그 모듈 구축
- 포모도로 기반 학습 타이머 및 학습 로그 저장 기능 구현
- 개인 학습 통계 대시보드 구축 (잔디밭 시각화, 주간 차트, 통계 카드)
- NextAuth 기반 사내 인증 체계(데모 계정 사전 정의) 구현
- 다크/라이트 테마 토글 및 localStorage 기반 사용자 설정 영속화
- PostgreSQL + Drizzle ORM 기반 데이터 영속 계층 구축
- Standalone 빌드 산출물 + PM2 기반 무중단 운영 환경 구성

### 2. 기능 요구사항 (SFR)

#### **SFR-001: 글 목록 조회 - 메인 페이지 카드 목록**

**요구사항 설명:**
- 메인 페이지(`/`)에 글 카드 형태의 목록을 표시한다.
- 각 카드는 다음 정보를 포함한다:
  - (선택) 커버 이미지
  - 태그 (React, Next.js, TypeScript, CSS, DB 등)
  - 읽기 예상 시간(분)
  - 제목
  - 한 줄 요약(excerpt)
  - 작성자 이름
  - 작성 날짜 (YYYY-MM-DD)
  - 좋아요 버튼 (♡ 아이콘 + 카운트)
- 글이 없는 빈 상태에서도 페이지가 정상 렌더링되며, 빈 상태 안내 메시지를 표시한다.
- 카드 hover 시 시각적 피드백(부상 효과 또는 색 변화)을 제공한다.
- 카드 본문 영역 클릭 시 해당 글의 상세 페이지로 이동한다.

---

#### **SFR-002: 글 검색 - 클라이언트 측 디바운스 검색**

**요구사항 설명:**
- 메인 페이지 글 목록 상단에 검색 입력란을 배치한다.
- 입력 시 **300ms 디바운스** 후 필터링을 적용한다.
- 검색 대상은 제목과 본문 요약(excerpt)이며, **대소문자 무관**으로 비교한다.
- 검색은 **클라이언트 측에서 수행**하며 서버 재요청을 발생시키지 않는다.
- 검색 결과가 0개일 경우 빈 상태 UI를 표시한다.

---

#### **SFR-003: 태그 필터링 - 카테고리별 글 분류**

**요구사항 설명:**
- 글 목록 상단에 태그 버튼 그룹을 배치한다.
  - "전체" 버튼 + 등록된 태그 종류별 버튼
- 태그 버튼 클릭 시 해당 태그의 글만 표시한다.
- "전체" 선택 시 모든 글을 표시한다.
- 현재 선택된 태그는 시각적으로 강조 표시한다.

---

#### **SFR-004: 글 정렬 - 다기준 정렬 옵션**

**요구사항 설명:**
- 드롭다운(select) 형태의 정렬 UI를 제공한다.
- 정렬 옵션 4종을 제공한다:
  - 최신순
  - 오래된순
  - 읽기 시간 짧은 순
  - 읽기 시간 긴 순
- 정렬 변경 시 즉시 결과에 반영한다.
- **검색 + 태그 필터 + 정렬이 동시 적용** 가능해야 한다.

---

#### **SFR-005: 글 상세 조회 - 단건 글 페이지**

**요구사항 설명:**
- URL 패턴: `/posts/[slug]` (slug는 URL 친화적 식별자)
- 상세 페이지 구성 요소:
  - 뒤로가기 링크 ("← 목록으로")
  - (선택) 큰 커버 이미지
  - 태그 + 읽기 시간
  - 제목
  - 작성자 + 날짜
  - 본문 전체 내용
- 페이지 로딩 중 **스켈레톤 UI**를 표시한다 (로딩 스피너 사용 금지).
- 존재하지 않는 slug 접근 시 **404 페이지를 자동 표시**한다.
- 글 조회 중 에러 발생 시 친근한 에러 UI를 표시하고 "다시 시도" / "목록으로" 두 버튼을 제공한다.

---

#### **SFR-006: 학습 타이머 - 포모도로 기반 시간 측정**

**요구사항 설명:**
- 메인 페이지 상단에 원형 진행률 표시기를 포함한 타이머 위젯을 배치한다.
- 목표 시간: **25분(1500초)** (포모도로 기법).
- 원형 게이지로 진행률을 시각화한다.
- 버튼 3종 제공: 시작, 정지, 초기화.
- 시작 후 매초 카운트가 증가하며 **MM:SS** 형식으로 표시한다.
- 상태 표시: "RECORDING" / "STANDBY".
- 저장 조건:
  - **로그인 상태 + 1분 이상 학습 후 정지** 시에만 DB에 학습 로그를 저장한다.
  - 비로그인 상태에서는 타이머는 사용 가능하나 DB 저장은 수행하지 않으며 안내 메시지를 표시한다.
- 저장 성공 시 "✓ N분 학습 기록됨" 메시지를 표시한다.
- 시간 설정 : 한국 표준시(UTC-9)를 기준으로 한다. 
- 시간 호환성 : DB의 timestamp 형식을 기준으로 Next.js의 형식을 통일한다. 둘의 형식이 맞는지 검증한다.

---

#### **SFR-007: 로그인 - 사전 정의 데모 계정 인증**

**요구사항 설명:**
- 로그인 페이지 경로: `/login`
- 아이디 + 비밀번호 입력 폼을 제공한다.
- 사전 하드코딩된 데모 사용자 **최소 3명** (alice, bob, kim)을 제공한다.
- 로그인 페이지 하단에 **데모 계정 정보 안내 박스**를 표시한다 (학습 편의).
- 로그인 실패 시 "아이디 또는 비밀번호가 올바르지 않습니다" 에러 메시지를 표시한다.
- 로그인 성공 시 **callbackUrl** 또는 메인 페이지로 자동 이동한다.
- 평문 비밀번호 비교를 허용하되, 코드 주석에 **운영 시 bcrypt 등 해시 필수**임을 명시한다.
- 인증 전략 : NextAuth인증전략 : JWT 기반
- JWT 생성 서명 : .env 파일의 NEXTAUTH_SECRET 키를 통해 생성 생성된 토큰으 별도 세션 없이 브라우저 쿠키에만 저장
- 보호 라우트와의 연동 : Middleware.ts 파일과 연동

---

#### **SFR-008: 글 작성 - 신규 글 발행**

**요구사항 설명:**
- 글 작성 페이지 경로: `/write`
- **비로그인 사용자가 직접 URL 접근 시 로그인 페이지로 자동 리다이렉트** (callbackUrl 자동 부착).
- 폼 필드:
  - 제목 (필수, 텍스트 입력)
  - 태그 (필수, 사전 정의된 옵션 중 선택)
  - 본문 (필수, 멀티라인 텍스트 영역)
- 버튼: "글 발행", "취소".
- 제출 중 버튼 비활성화 + "저장 중..." 표시.
- 제출 성공 시 동작:
  - 글 영구 저장
  - 새 글 상세 페이지로 자동 이동
  - 메인 페이지 글 목록에 즉시 반영 (캐시 무효화)
- 자동 채워지는 필드:
  - 작성자 = 로그인 사용자의 이름
  - 작성 날짜 = 오늘 (YYYY-MM-DD)
  - slug = 제목에서 자동 생성 (소문자, 공백→하이픈, 특수문자 제거)
  - 읽기 시간 = 본문 길이 기반 추정 (약 500자당 1분)
  - excerpt = 본문 첫 100자

---

#### **SFR-009: 좋아요 버튼 - 클라이언트 측 토글**

**요구사항 설명:**
- 각 글 카드에 ♡ 아이콘 + 카운트 형태의 좋아요 버튼을 배치한다.
- 클릭 시 토글 동작:
  - ♡ → ♥ (또는 색 변화)
  - 카운트 ±1
- **좋아요 상태는 영구 저장하지 않는다** (페이지 새로고침 시 초기화) — 의도된 단순화.
- 좋아요 클릭 시 카드 본문 영역 클릭(상세 이동)이 발생하지 않도록 이벤트 전파를 차단한다.
- 다른 카드의 좋아요 상태에 영향을 주지 않는다.
- **정렬 변경 시 좋아요 상태가 각 글을 따라가야 한다** (위치 기반이 아닌 글 ID 기반).

#### **SFR-010: 개인 대시보드 - 학습 통계 시각화**

**요구사항 설명:**
- 대시보드 경로: `/dashboard`
- 비로그인 사용자 접근 불가 (로그인 페이지로 리다이렉트).
- 다른 사용자의 대시보드 조회 불가 (자신의 데이터만 표시).
- 구성 영역:
  - **통계 카드 3개:**
    - 누적 학습 시간 (예: "12h 30m")
    - 이번 주 학습 시간 (월요일~현재, 예: "3h 15m")
    - 연속 학습 일수 / Streak (예: "5일")
      - streak > 0: "잘 하고 계세요 🔥"
      - streak = 0: "오늘부터 시작!"
  - **잔디밭 (Contribution Graph):**
    - 격자: 26주 × 7일 = 182칸
    - 마지막 칸 = 이번 주 토요일(또는 오늘 속한 주 마지막 날)
    - 각 칸 = 하루, 색 강도로 학습량 표현
    - 활동량 → 색 강도 매핑 (5단계):
      - 0: 학습 없음 — 어두운 회색
      - 1: 30분 미만 — 매우 흐린 시안
      - 2: 30분~1시간 — 흐린 시안
      - 3: 1시간~2시간 — 중간 시안
      - 4: 2시간 이상 — 진한 시안
    - 미래 날짜: opacity 낮춤
    - hover 툴팁: "YYYY-MM-DD — N분 학습"
    - 하단 범례: "적음 [□□□□□] 많음"
    - 반응형: 가로 폭이 좁으면 가로 스크롤 지원
    - **Hydration mismatch 방지**: 클라이언트 컴포넌트 + mounted 패턴 또는 일관된 로컬 타임존 YYYY-MM-DD 키 사용
  - **주간 차트:**
    - 최근 7일의 일별 학습 시간 막대 차트
    - X축: 요일 라벨 (일/월/화/수/목/금/토)
    - Y축: 분 단위
    - 데이터 없는 날도 0 막대로 표시
    - hover 툴팁 제공
    - 막대 그라데이션 (위→아래 진함→흐림)

#### **SFR-011: 테마 토글 - 다크/라이트 모드 전환**

**요구사항 설명:**

### 4. 보안 요구사항 (SER)

#### **SER-001: 인증·인가 - 다층 방어 원칙**

  - 2차: **Server Action 또는 Server Component** 내부에서 세션 재검증
- 비로그인 사용자가 보호 라우트 접근 시 `/login?callbackUrl=...` 형태로 리다이렉트한다.

**요구사항 설명:**

- `.env` 파일은 git 추적에서 제외한다 (`.gitignore` 적용).
- `.env.example` 파일은 git에 포함하되 실제 값은 비워두고 키만 정의한다.
- 필수 환경변수:
  - `DATABASE_URL`
  - `NEXTAUTH_SECRET`
  - `NEXTAUTH_URL`

#### **SER-003: 입력 검증 - 사용자 입력 처리**

**요구사항 설명:**
- 사용자 입력은 항상 검증 후 사용한다.
- Server Action 진입 시 세션 존재 여부 + 입력 필수 필드 + 타입 검증을 수행한다.
- 빈 문자열, 과도하게 긴 입력 등 비정상 입력에 대한 방어 로직을 포함한다.

#### **SER-004: 에러 메시지 안전성 - 민감 정보 노출 방지**

**요구사항 설명:**

### 5. 품질 요구사항 (QAR)

#### **QAR-001: 코드 품질 - TypeScript / ESLint**

**요구사항 설명:**

#### **QAR-002: 코드 구조 - 단일 책임 원칙**

**요구사항 설명:**

- **함수 시그니처를 통한 추상화** 적용 — 데이터 출처가 바뀌어도(메모리 → DB) 호출 측 코드는 불변이어야 한다.

#### **QAR-003: 에러 처리 - 사용자 친화적 경계**

**요구사항 설명:**
- 글 상세 페이지에 `error.tsx` 로 에러 경계를 자동 적용한다.
- 글 상세 페이지에 `loading.tsx` 로 로딩 UI를 자동 적용한다.
- 존재하지 않는 글 접근 시 `notFound()` 호출로 404 페이지로 자동 이동시킨다.
- 사용자에게는 친근한 에러 메시지만 표시하고 기술적 스택 트레이스는 노출하지 않는다.

#### **QAR-004: 접근성 - 시맨틱 / 키보드 / 색대비**

**요구사항 설명:**

- 키보드만으로 주요 기능 사용 가능.
- 색 대비를 충분히 확보한다.

---

#### **QAR-005: Hydration 안정성 - SSR/CSR 일관성**

**요구사항 설명:**
- `new Date()`, `Date.now()`, `toISOString()` 등 시간 의존 코드를 SSR 시점에 사용할 경우 **Hydration mismatch**를 발생시키지 않도록 한다.
- 대시보드 잔디밭 등 시간 의존 컴포넌트는 다음 중 하나를 적용한다:
  - 방법 A (권장): 클라이언트 컴포넌트 + `useState/useEffect` mounted 패턴
  - 방법 B: 모든 날짜 키를 로컬 타임존 기준 YYYY-MM-DD로 일관 생성
- 브라우저 콘솔 및 서버 콘솔에 비정상 에러가 없어야 한다.

- **운영 망**: 사내망 (외부 인터넷 의존성 최소화)
- **프로세스 관리**: PM2 기반 무중단 운영, 로그 자동 회전
- **빌드 형태**: Next.js Standalone 빌드 (`output: "standalone"`)
- **환경변수 관리**: `.env` (실제값, git 제외) + `.env.example` (키만, git 포함)
- **데이터 영속성**: PostgreSQL 17, Drizzle ORM, `npm run db:push`로 스키마 동기화
- **시드 데이터**: `npm run seed` 명령으로 초기 글 6개 이상 일괄 입력
- **외부 의존**: 커버 이미지 placeholder(`picsum.photos`)만 외부 호출, 그 외 의존성 없음
